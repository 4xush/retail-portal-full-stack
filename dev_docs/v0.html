Retail portal full-stack design reference

System architecture

Recommended approach: Modular monolith → easy microservice extraction later

For a 2–4 day hackathon, a clean monolith with well-separated modules beats premature microservices. Use folder-based separation so any module can be extracted later without rewriting the API contracts.

High-level request flow

1
Browser → React SPA

User interacts → React Query fires API call with JWT in Authorization header

2
Express router

auth middleware validates JWT / API key → attaches req.user with role

3
Route → Controller → Service

Controllers are thin (parse req, call service, return res). All business logic lives in services.

4
Service → Mongoose → MongoDB

Service layer calls Mongoose models. Aggregation pipelines handle complex queries (combos, order history with product data).

5
Response → React Query cache

Successful response updates client cache. Stale-while-revalidate for product listings.

Trade-off notes

NoSQL (MongoDB) vs SQL

MongoDB wins here: product catalog with variable fields (combos, add-ons, tax%), nested category trees, and flexible order snapshots all map naturally to documents. Use SQL if you're more comfortable with it — the eval rarely cares which you pick as long as you can explain why.

Data flows for key features

Auth flow (signup / login)

1
POST /api/auth/signup

Validate email uniqueness → bcrypt hash password (rounds: 12) → save User doc → return JWT + refresh token

2
POST /api/auth/login

Find user by email → bcrypt.compare → sign JWT (15m expiry) + refresh token (7d, stored in httpOnly cookie) → return access token in body

3
Every protected request

Authorization: Bearer {token} → middleware verifies signature + expiry → req.user = decoded payload

4
Token refresh

POST /api/auth/refresh → read httpOnly cookie → verify refresh token → issue new access token

API Key flow: Admin can generate API keys (stored hashed in DB). Keys are passed as X-API-Key header. Middleware checks this before JWT. Useful for Postman demo.

Product listing & fuzzy search

1
Home page load

GET /api/categories?include=products&limit=8 — returns categories each with top N products (aggregation $lookup). Lazy load more per category on scroll.

2
Fuzzy search

MongoDB text index on {name, description, tags}. Query: db.products.find({$text:{$search: q}, score: {$meta: 'textScore'}}).sort({score}). For better fuzzy matching, add a simple regex fallback: /query/i on name field.

3
Pagination

Cursor-based for infinite scroll (lastId param). Offset-based for admin tables (page + limit params). Never load all records — default limit: 20.

4
Category filter

GET /api/products?category=cat_id&page=1&limit=20. Category ID indexed in products collection.

Cart / combos / add-ons

1
Cart stored in DB (not just localStorage)

One Cart doc per user. CartItem embeds productId, qty, selectedAddOns[], selectedComboId, snapshotPrice (price at time of add).

2
Combo logic

ComboGroup doc: {name, products: [{productId, qty}], discountType: 'percent'|'flat', discountValue}. When user selects combo, frontend sends comboId; backend re-validates combo price server-side before order creation.

3
Add-ons

Product schema has addOns: [{name, price, required: bool}]. Frontend renders add-on selector. Add-on selections stored in cart item.

4
Price calculation

Always recalculate total on backend at checkout. Never trust client-sent price. Formula: (basePrice + addOnTotal) × qty × (1 + taxRate) × (1 - discountRate).

Critical edge case: Product price or stock changes after item added to cart. Snapshot the price at add-to-cart time. At checkout, warn user if price changed and require confirmation. Deduct stock atomically with findOneAndUpdate + $inc:{stock: -qty} + check stock ≥ 0 in query filter.

Order history & re-ordering

1
Order creation

Atomic: decrement stock for all items → create Order doc with full product snapshot (name, price, image at time of order) → clear cart. If any stock check fails, rollback with session transactions.

2
Order history page

GET /api/orders?page=1&limit=10. Returns orders sorted by createdAt desc. Each order embeds product snapshot so history is accurate even if product is deleted later.

3
Re-order flow

POST /api/orders/:id/reorder → fetch original order → for each item, check current stock & price → add to cart (with updated prices, flagging changes) → redirect to cart.

Indexing strategy

  • Text index on products: {title, description, tags} — enables $text search
  • Compound index on products: {categoryId: 1, isAvailable: 1, createdAt: -1} — category page query
  • Index on orders: {userId: 1, createdAt: -1} — order history
  • Unique index on users.email — enforced at DB level
  • TTL index on carts.updatedAt: 30 days expiry

Common mistakes & bonus impressors

Mistakes candidates make

🔴

Trusting client-sent prices. Always recalculate totals server-side at checkout. Evaluators test this by modifying the request.

🔴

Not handling race conditions on stock. Two users buying the last item simultaneously. Use atomic findOneAndUpdate with stock check in the filter, not in application code.

🔴

Returning passwordHash in API responses. Always use .select('-passwordHash') or transform via toJSON. Evaluators check this in network tab.

🔴

No error handling middleware. Unhandled promise rejections crash the server. Add a global Express error handler and async wrapper utility.

🔴

Storing full product data in order items by reference. If product is deleted/edited, order history breaks. Always snapshot title, price, image into the order document.

🟡

No input validation. Use express-validator or Zod on every endpoint. Evaluators send malformed requests.

🟡

Committing .env files. Use .env.example with placeholder values. Real secrets in .env (gitignored). Your GitHub Actions should use repo secrets.

🟡

N+1 query in product listing. Don't fetch category then loop products per category. Use $lookup aggregation in a single query.

🟡

No pagination defaults. Always enforce max limit (e.g. 100) even if client sends limit=99999.

🟡

Admin routes with no role check. JWT alone doesn't mean admin. Middleware must check req.user.role === 'admin'.

Bonus features that impress

  • MongoDB Atlas Search — upgrade fuzzy search to Lucene-powered full-text with typo tolerance. Takes ~30 mins.
  • Postman Newman in CI — add a GitHub Actions step that runs your Postman collection against the staging API. Shows testing maturity.
  • Optimistic UI updates — cart updates feel instant (React Query's optimisticUpdate). Shows frontend polish.
  • Skeleton loaders — instead of spinners, show content-shaped loading states. Huge UX impression.
  • Rate limiting — express-rate-limit on auth routes (5 req/min). Shows security awareness.
  • Stock history audit log — every stock change tracked with who, when, why. Shows domain thinking.
  • KFC-style sticky category navbar — as user scrolls, active category highlights. Classic e-com pattern.
  • Swagger/OpenAPI docs — auto-generated from swagger-jsdoc. Evaluators can explore the API interactively.

Security checklist

  • helmet() — sets secure HTTP headers
  • cors() — restrict origins to your frontend URL in production
  • express-rate-limit — auth endpoints especially
  • mongoSanitize — prevent NoSQL injection via user inputs
  • JWT in Authorization header (not localStorage for refresh token)
  • Signed Cloudinary upload URLs — never expose your API secret
  • HTTPS enforced in production
